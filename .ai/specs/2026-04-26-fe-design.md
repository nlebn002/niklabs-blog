# Spec: Frontend Design System — shadcn/ui + Tailwind v4 + Dark Mode

**Date:** 2026-04-26  
**Scope:** Frontend only (`frontend/`)

---

## Goal

Migrate the frontend to Tailwind CSS v4 and install shadcn/ui. Add a proper dark/light mode toggle persisted in localStorage. All theming lives in a single global CSS file — no JS config needed for colours or tokens.

---

| Thing | Current |

## 1. Tailwind v4 Upgrade

### Package changes

```bash
# Remove
npm uninstall tailwindcss autoprefixer

# Add
npm install -D tailwindcss@latest @tailwindcss/postcss@latest
```

`autoprefixer` is no longer needed in v4 — remove it from devDependencies.

### PostCSS config

Replace (or create) `postcss.config.js`:

```js
export default {
  plugins: {
    "@tailwindcss/postcss": {}
  }
};
```

### tailwind.config.js

Delete it entirely. All config moves into `src/styles/index.css`.

### Global CSS — `src/styles/index.css`

Tailwind v4 uses `@import "tailwindcss"` and a `@theme {}` block instead of a JS config. Existing CSS variable names (`--background`, `--foreground`, etc.) are preserved — they are the shadcn convention and must not be renamed.

Dark mode strategy: `class` — toggle `.dark` on `<html>`. In v4 this is declared as a custom variant.

**Drop the `sage` and `slate` themes.** Keep only the light default and `.dark`.

Run after the Tailwind v4 upgrade is confirmed working:

```bash
npx shadcn@latest init
```

CLI answers:
- Style: **Default**
- CSS file: `src/styles/index.css`
- CSS variables: **Yes**
- Tailwind config path: leave empty (v4 mode, no JS config)
- Components alias: `@/components`
- Utils alias: `@/utils/cn`

## 3. Dark / Light Mode

### Flash prevention — `index.html`

Add as the **first child of `<head>`**, before any `<link>` or `<style>`:

```html
<script>
  (function () {
    var stored = localStorage.getItem("theme");
    var prefersDark = window.matchMedia("(prefers-color-scheme: dark)").matches;
    if (stored === "dark" || (!stored && prefersDark)) {
      document.documentElement.classList.add("dark");
    }
  })();
</script>
```

### ThemeProvider

New feature folder: `src/features/theme/`

```
src/features/theme/
  model/
    theme-context.tsx   # React context + ThemeProvider component
    use-theme.ts        # useTheme() hook
  ui/
    theme-toggle.tsx    # icon button (sun / moon)
```

**`theme-context.tsx`** — manages state, syncs to localStorage and `document.documentElement.classList`:

```tsx
type Theme = "light" | "dark";
type ThemeContextValue = {
  theme: Theme;
  setTheme: (t: Theme) => void;
  toggle: () => void;
};
```

On mount: read `localStorage.getItem("theme")`. If absent, read `window.matchMedia("(prefers-color-scheme: dark)").matches`. Subscribe to the media query change event and update theme automatically only when no stored preference exists.

On theme change: write to localStorage and add/remove `.dark` on `document.documentElement`.

**`use-theme.ts`** — thin hook over the context.

**`theme-toggle.tsx`** — shadcn Button with `variant="ghost"` and `size="icon"`. Renders a sun icon in dark mode and a moon icon in light mode. No text label. Place in the far-right position of the `SiteHeader` nav row.

### App.tsx

```tsx
<QueryClientProvider client={queryClient}>
  <ThemeProvider>
    <BrowserRouter>
      <AppRouter />
    </BrowserRouter>
  </ThemeProvider>
</QueryClientProvider>
```

## 5. Acceptance Criteria

- `npm run build` completes with zero TypeScript errors.
- Light and dark mode render correctly; no white flash on hard reload in either mode.
- Toggling theme persists across page reloads.
- System preference is respected when no stored choice exists; switching OS preference updates the app only if the user has not made a manual choice.
- All existing pages (home, post detail, post editor, sign-in, change-password) render without visual regressions in both modes.
- All colour/radius/shadow changes are made exclusively in `src/styles/index.css`.
- `buttonVariants` is importable from `@/components/ui/button` for use with `<Link className={buttonVariants({ variant: "default" })} />`.
- `radix-ui` package remains (Dialog already imports from it); shadcn components that overlap use their own Radix primitives — no duplication issues expected.
