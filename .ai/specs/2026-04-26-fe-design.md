# Spec: Frontend Design System Modernization

**Date:** 2026-04-26  
**Scope:** Frontend only (`frontend/`)

---

## Goal

Modernize the frontend around current Tailwind CSS v4 and shadcn/ui guidance for a Vite React app.

Use Tailwind's official Vite plugin integration, migrate all design tokens into `src/styles/index.css`, replace valuable hand-rolled UI primitives with shadcn-compatible components, and move theme handling out of the layout shell into a reusable `light | dark | system` theme feature.

The result should be modern, maintainable, easy for AI tools to reason about, and compatible with the app's existing call sites.

## Implementation Plan

### 1. Upgrade Tailwind to v4 using the Vite plugin

Use the current official Vite path instead of the older PostCSS-centered setup.

Package changes:

```bash
npm uninstall autoprefixer
npm install -D tailwindcss@latest @tailwindcss/vite@latest
```

`tailwindcss` remains a dev dependency. `autoprefixer` is no longer needed for this app's Tailwind setup.

Update `vite.config.ts`:

```ts
import tailwindcss from "@tailwindcss/vite";
```

Add `tailwindcss()` to the Vite `plugins` array next to `react()`.

Remove obsolete Tailwind v3 wiring:

- Delete `tailwind.config.js`.
- Delete `postcss.config.js` if it only contains Tailwind/autoprefixer wiring.

Do not add `@tailwindcss/postcss`; this spec targets the Vite plugin integration.

### 2. Convert global CSS to Tailwind v4

`src/styles/index.css` becomes the single source of truth for tokens, theme mapping, fonts, and app-level utilities.

Required structure:

```css
@import "tailwindcss";
@import url("https://fonts.googleapis.com/css2?family=Fraunces:opsz,wght@9..144,600;9..144,700&family=Manrope:wght@400;500;600;700;800&display=swap");

@variant dark (&:is(.dark *));
```

Then define:

- `@layer base` with `:root` light tokens and `.dark` dark tokens.
- `@theme` with Tailwind v4 token mappings.
- Existing app utilities: `.bg-site`, `.font-display`, `.editorial-rule`, `.article-prose`.

Token rules:

- Convert token values from HSL tuples to direct `oklch(...)` CSS values.
- Preserve shadcn token names such as `--background`, `--foreground`, `--card`, `--primary`, `--border`, `--ring`, and `--radius`.
- Drop `[data-theme="default"]`, `[data-theme="sage"]`, `[data-theme="slate"]`, and `[data-theme="dark"]`.
- Keep only `:root` for light mode and `.dark` for dark mode.
- All color, radius, font, and shadow changes must live in `src/styles/index.css`.

Example theme mapping:

```css
@theme {
  --color-background: var(--background);
  --color-foreground: var(--foreground);
  --color-card: var(--card);
  --color-card-foreground: var(--card-foreground);
  --color-popover: var(--popover);
  --color-popover-foreground: var(--popover-foreground);
  --color-primary: var(--primary);
  --color-primary-foreground: var(--primary-foreground);
  --color-secondary: var(--secondary);
  --color-secondary-foreground: var(--secondary-foreground);
  --color-muted: var(--muted);
  --color-muted-foreground: var(--muted-foreground);
  --color-accent: var(--accent);
  --color-accent-foreground: var(--accent-foreground);
  --color-destructive: var(--destructive);
  --color-destructive-foreground: var(--destructive-foreground);
  --color-border: var(--border);
  --color-input: var(--input);
  --color-ring: var(--ring);

  --radius-sm: calc(var(--radius) - 4px);
  --radius-md: calc(var(--radius) - 2px);
  --radius-lg: var(--radius);
  --radius-xl: calc(var(--radius) + 4px);

  --shadow-card: 0 30px 80px oklch(from var(--shadow-color) l c h / 0.14);
  --shadow-soft: 0 22px 56px oklch(from var(--shadow-color) l c h / 0.10);

  --font-sans: "Manrope", sans-serif;
  --font-display: "Fraunces", serif;
}
```

Audit class names after the token conversion. Existing HSL-specific arbitrary classes such as `bg-[hsl(var(--card))]` must become token utilities such as `bg-card`.

### 3. Update shadcn setup

Keep the existing `frontend/components.json`, but update it for Tailwind v4 before adding components.

Expected shadcn configuration:

- Style: `default`
- TSX: `true`
- CSS file: `src/styles/index.css`
- CSS variables: `true`
- Tailwind config path: empty or omitted, because Tailwind v4 config lives in CSS
- Components alias: `@/components`
- UI alias: `@/components/ui`
- Utils alias: `@/utils`
- RSC: `false`

Install or refresh components with the latest CLI:

```bash
npx shadcn@latest add button
npx shadcn@latest add input
npx shadcn@latest add textarea
npx shadcn@latest add dialog
npx shadcn@latest add alert
npx shadcn@latest add badge
npx shadcn@latest add dropdown-menu
npx shadcn@latest add separator
npx shadcn@latest add tooltip
```

Allow CLI-added dependencies such as:

- `class-variance-authority`
- `lucide-react`
- individual `@radix-ui/react-*` packages needed by selected shadcn components

The existing `radix-ui` package may remain unless it becomes unused after migration and build verification confirms it can be removed.

### 4. Preserve compatibility for existing call sites

The implementation should modernize internals without forcing broad, unnecessary call-site rewrites.

#### Button

Replace `src/components/ui/button.tsx` with a shadcn-compatible Button implementation.

Export:

- `Button`
- `buttonVariants`
- `buttonStyles`

`buttonVariants` should support the shadcn object API:

```tsx
buttonVariants({ variant: "default", size: "default" });
```

`buttonStyles` must remain as a compatibility wrapper for current callers:

```tsx
buttonStyles("primary", "px-6", "lg");
```

Legacy variant mapping:

| Existing | New |
|---|---|
| `primary` | `default` |
| `danger` | `destructive` |
| `secondary` | `secondary` |
| `ghost` | `ghost` |
| `outline` | `outline` |

Existing sizes `default`, `sm`, `lg`, and `icon` must keep working.

#### Alert

Replace the visual implementation with shadcn Alert styling, but keep the current app-level API:

```tsx
<Alert title="Could not sign in" message={error.message} />
<Alert title="Password updated" message="..." variant="success" />
```

Supported variants:

- `error`
- `success`

#### Dialog

Use shadcn-compatible Dialog internals, but preserve existing export names:

- `DialogRoot`
- `DialogTrigger`
- `DialogPortal`
- `DialogOverlay`
- `DialogClose`
- `DialogContent`
- `DialogHeader`
- `DialogFooter`
- `DialogTitle`
- `DialogDescription`

Existing callers such as delete confirmation dialogs must not need changes.

#### Input and Textarea

Replace with shadcn-compatible components while keeping the existing import paths and default ergonomics.

#### Panel

Keep `panel.tsx` as a custom layout primitive. It is not replaced by shadcn.

### 5. Move theme behavior into a feature module

Create:

```text
src/features/theme/
  model/
    theme-context.tsx
    use-theme.ts
  ui/
    theme-toggle.tsx
```

Theme model:

```ts
type Theme = "light" | "dark" | "system";
type ResolvedTheme = "light" | "dark";

type ThemeContextValue = {
  theme: Theme;
  resolvedTheme: ResolvedTheme;
  setTheme: (theme: Theme) => void;
};
```

Behavior:

- Persist selected theme in localStorage key `niklabs-theme`.
- Default to `system` when no stored value exists.
- If stored value is invalid, fall back to `system`.
- Apply only `.light` or `.dark` to `document.documentElement`.
- Remove legacy `data-theme` writes.
- When `theme === "system"`, follow `window.matchMedia("(prefers-color-scheme: dark)")`.
- React to OS preference changes only while `theme === "system"`.

Wrap the app in `ThemeProvider`:

```tsx
<QueryClientProvider client={queryClient}>
  <ThemeProvider defaultTheme="system" storageKey="niklabs-theme">
    <BrowserRouter>
      <AppRouter />
    </BrowserRouter>
  </ThemeProvider>
</QueryClientProvider>
```

### 6. Prevent theme flash

Add this as the first child of `<head>` in `frontend/index.html`, before any stylesheet or preload:

```html
<script>
  (function () {
    var storageKey = "niklabs-theme";
    var stored = localStorage.getItem(storageKey);
    var theme = stored === "light" || stored === "dark" || stored === "system" ? stored : "system";
    var systemDark = window.matchMedia("(prefers-color-scheme: dark)").matches;
    var resolved = theme === "system" ? (systemDark ? "dark" : "light") : theme;
    document.documentElement.classList.remove("light", "dark");
    document.documentElement.classList.add(resolved);
  })();
</script>
```

### 7. Theme toggle UI

Replace the current theme button in `SiteShell` with `ThemeToggle`.

Use shadcn primitives:

- `Button`
- `DropdownMenu`
- `DropdownMenuTrigger`
- `DropdownMenuContent`
- `DropdownMenuItem`

Use `lucide-react` icons:

- `Sun`
- `Moon`
- `Monitor`

Expected behavior:

- Button is `variant="ghost"` and `size="icon"`.
- Button icon reflects `resolvedTheme`: sun for light, moon for dark.
- Dropdown options: Light, Dark, System.
- Active option is visibly marked.
- No visible explanatory text outside the dropdown.
- Accessible labels are present for the trigger and options.

Place the toggle in the right side of the existing `SiteShell` nav row.

## Acceptance Criteria

- `npm run build` completes with zero TypeScript errors.
- Tailwind v4 is integrated through `@tailwindcss/vite`.
- `tailwind.config.js` is removed.
- `postcss.config.js` is removed if it only contained old Tailwind/autoprefixer wiring.
- `components.json` is compatible with shadcn + Tailwind v4.
- Light, dark, and system theme modes work.
- Stored theme persists across reloads using `niklabs-theme`.
- No white flash appears on hard reload in dark mode.
- System preference is respected when theme is `system`.
- OS preference changes update the app only while theme is `system`.
- No code writes `data-theme`.
- `buttonVariants` is importable from `@/components/ui/button`.
- Existing `buttonStyles(...)` callers continue working.
- Existing `DialogRoot` callers continue working.
- Existing `<Alert title message variant />` callers continue working.
- All existing pages render correctly in light and dark mode:
  - home
  - post detail
  - post editor
  - sign-in
  - change-password
- All color, radius, font, and shadow token changes live in `src/styles/index.css`.

## Out of Scope

- Backend changes.
- API contract changes.
- Orval client regeneration.
- New pages.
- Reintroducing `sage` or `slate` as selectable themes.
- Large visual redesign beyond the design-system modernization and theme behavior.
